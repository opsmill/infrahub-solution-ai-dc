## Key Patterns Demonstrated

This solution illustrates five architectural patterns applicable to infrastructure automation:

### 1. Hierarchical Generation

Generate bottom-up: parents complete before children run.

```
Fabric (super-spines) → Pod (spines) → Rack (leafs)
       ↓               ↓               ↓
  Resource pools   Inherit pools    Use pools
   created        created by        from parent
                   parent
```

**When to use**: Multi-level infrastructure where each level depends on previous.

**Infrahub mechanism**: GeneratorTarget mixin + checksums trigger child generators.

### 2. Checksum-Driven Cascading

Design change → checksum update → child generator triggers automatically.

**Alternative**: Manual trigger orchestration (fragile, error-prone).

**Why this works**: Checksums hash all affected objects. Change → hash changes → detected automatically.

**Extension**: Add hooks (webhooks, Kafka) to notify external systems when checksum changes.

### 3. Deterministic Algorithms

Index-based formulas ensure predictable, collision-free resource allocation.

```
Pod index → interface ranges
Rack index → interface windows
Device index → specific port within window
```

**Why important**: Allows you to manually predict cables before deployment. Run formula yourself, match against generated plan.

**Extension**: Add validation: post-generation, verify algorithm constraints (no overlaps, full utilization).

### 4. Resource Pool Hierarchy

Infrastructure pools nest: fabric → pod → device.

```
FabricSupernetPool (/8)
  ├─ Fabric-A supernet (/19)
  │  ├─ Fabric-A loopback pool (/28)
  │  └─ Fabric-A prefix pool
  │     ├─ Pod-A supernet (/19)
  │     │  └─ Pod-A loopback pool (/27)
  │     ├─ Pod-B supernet (/19)
  │     └─ Pod-C supernet (/19)
```

**Benefit**: Each level allocates what it needs from parent. Prevents overlap, enables scoping.

**Extension**: Add quota enforcement: "Fabric-A can allocate max 100 devices," check before creating.

### 5. Template Composition

Templates define structure, profiles define properties. Compose them:

```
Device Template
  ├─ Interface[1-27] + profile-spine-downlink
  ├─ Interface[28-31] + profile-spine-uplink
  └─ Loopback0 + profile-loopback

Profile properties:
  ├─ MTU, status, description
  └─ VRF, ACLs (extensible)
```

**Benefit**: Change profile → all interfaces using it update automatically.

**Extension**: Add conditionals: "If device_type = S9250, use profile-flex-ports; if S9260, use profile-enhanced-ports."

## Extending the Solution

### Add New Rack Types

**Scenario**: You have compute racks and storage racks. Now add GPU racks.

**Implementation**:

1. Create new device template:

```yaml
CoreObjectTemplate:
  - name: leaf-switch-gpu
    description: "GPU rack leaf with 16 HBM ports"
    nodes:
      - kind: NetworkDevice
        relationships:
          interfaces:
            - name: Ethernet[1-24]
              profile: profile-gpu-downlink
            - name: Ethernet[25-28]
              profile: profile-gpu-uplink
```

2. Update schema:

```yaml
- name: Rack
  attributes:
    - name: rack_type
      choices:
        - compute
        - storage
        - gpu  # ← Add this
```

3. Update RackGenerator template selection:

```python
template_map = {
    "compute": "leaf-switch-compute",
    "storage": "leaf-switch-storage",
    "gpu": "leaf-switch-gpu",  # ← Add this
}
template_id = template_map[rack_type]
```

4. Create GPU racks with `rack_type: gpu` in data files.

**Result**: New rack type works seamlessly. Cabling algorithm unchanged.

### Add Validation

**Scenario**: Before generation, validate design is sensible.

**Implementation**:

```python
class FabricValidationGenerator(InfrahubGenerator):
    async def generate(self, data: dict) -> None:
        """Validate fabric design before generation"""
        fabric = data.network_fabric.edges[0].node
        errors = []

        # Check amount_of_super_spines divisible by 2 (redundancy)
        if fabric.amount_of_super_spines.value % 2 != 0:
            errors.append("Super-spines must be even (redundancy)")

        # Check template exists
        if not fabric.super_spine_switch_template.node:
            errors.append("Super-spine template not set")

        # Check IP pools available
        pools = await client.filters(
            kind=CoreIPPrefixPool,
            name__value="fabric-supernet-pool"
        )
        if not pools:
            errors.append("Fabric supernet pool not found")

        if errors:
            raise ValueError(f"Design validation failed: {'; '.join(errors)}")

        # If valid, signal "ready for generation"
        fabric.validation_status = "passed"
        await fabric.save(allow_upsert=True)
```

Run before FabricGenerator. Prevent cascading errors.

## Design Decisions Explained

### Why Checksums vs Manual Orchestration?

**Checksums** (current):
- Pro: Automatic, change → regenerate
- Pro: Works with event triggers
- Con: Hash logic complex

**Manual orchestration** (alternative):
- Pro: Simple, explicit
- Con: Teams forget to trigger child
- Con: Drift between design and reality
- Con: Doesn't scale (10 fabrics = 10 × 3 = 30 manual triggers)

**Decision**: Checksums win for scalability.

### Why Hierarchical Schema vs Flat Device List?

**Hierarchical** (current):
- Pro: Scoped automation (pod-level changes don't affect others)
- Pro: Resource pooling naturally nested
- Pro: Mirrors physical organization
- Con: Deeper codebase, more relationships

**Flat** (alternative):
- Pro: Simple schema
- Con: Can't apply pod-level configs easily
- Con: Resource conflicts across fabrics

**Decision**: Hierarchical better for multi-fabric deployments.

### Why Templates vs Full Device Definitions?

**Templates** (current):
- Pro: Change template → all instances update
- Pro: Reusable across pods/racks
- Pro: Small YAML footprint
- Con: Indirection (need to understand templates)

**Full definitions** (alternative):
- Pro: Self-contained, visible
- Con: Duplication (12 spines = 12 identical definitions)
- Con: Change = edit 12 files

**Decision**: Templates win for maintainability.

### Why Index-Based Cabling vs Manual Port Assignment?

**Index-based** (current):
- Pro: Scales to any pod/rack count
- Pro: Collision-free mathematically
- Pro: Predictable (formula works forever)
- Con: Requires understanding algorithms

**Manual assignment** (alternative):
- Pro: Visual, understandable
- Con: Doesn't scale (1000 racks = 1000 manual assignments)
- Con: Error-prone
- Con: Hard to evolve

**Decision**: Index-based essential for scale.

## Performance Characteristics

Based on solution performance:

| Operation | Time | Scaling |
|-----------|------|---------|
| FabricGenerator (6 SS) | ~10s | Linear with SS count |
| PodGenerator (4 spines) | ~15s | Linear with spines × SS connections |
| RackGenerator (2 leafs) | ~10s | Linear with leafs × spine connections |
| Full fabric (6 SS, 3 pods, 10 racks) | ~5 min | Linear with total devices |
| Cabling plan generation | ~2-3s | Linear with link count |

## Customization Points

Key areas where you'll extend this solution:

### 1. Generator Logic

Add custom generator for your infrastructure:

```python
class KeepAliveMonitoringGenerator(InfrahubGenerator):
    """Create monitoring configs for all devices"""
    async def generate(self, data: dict) -> None:
        devices = await client.filters(kind=NetworkDevice)
        for device in devices:
            config = f"""
            snmp-server community {device.name}
            snmp-server contact infrastructure@company.com
            """
            device.monitoring_config = config
            await device.save()
```

```python
class CablingValidation:
    """Validate all cables are accounted for"""
    async def validate(self, fabric):
        expected_cables = (
            fabric.amount_of_super_spines * pods * spines +
            pods * racks * leafs * spines
        )
        actual_cables = await count_links()
        assert expected_cables == actual_cables
```

## Production Deployment

### Checklist Before Going Live

- [ ] **Validation**: Run FabricValidationGenerator (or custom validator)
- [ ] **Dry-run**: Test generators in non-prod environment
- [ ] **Cabling plan**: Export CSV, validate with network team
- [ ] **Compatibility**: Verify device types are in use (S9250, etc.)
- [ ] **IP addressing**: Confirm supernet pool size adequate
- [ ] **Monitoring**: Set up alerts on generator failures
- [ ] **Documentation**: Update runbooks with generator details
- [ ] **Rollback plan**: How to handle failed generation?
- [ ] **Change management**: Governance for design changes

### Monitoring Generated Infrastructure

Set up alerts on:

1. **Generator failures**: Alert on generation time >2×baseline
2. **Checksum mismatches**: Pod/rack checksum != expected
3. **Incomplete generation**: Device count != expected
4. **Link failures**: Unconnected interfaces
5. **IP allocation drift**: Duplicate addresses detected

## Troubleshooting Guide

| Issue | Cause | Fix |
|-------|-------|-----|
| FabricGenerator fails | Super-spine template missing | Verify template exists in objects |
| PodGenerator fails | Super-spines don't exist | Run FabricGenerator first |
| RackGenerator fails | Spines don't exist | Run PodGenerator first |
| Cabling collisions | Index formula error | Verify rack/pod indexes sequential |
| IP allocation errors | Pool size too small | Increase prefix pool size |
| Slow generation | Infrahub overloaded | Check Infrahub resource usage |
